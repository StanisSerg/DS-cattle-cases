## Архитектурное давление

Проект `DS-cattle-cases` решает задачу: хранить сырые кейсы с ферм так, чтобы из них позже выращивались обобщённые решения (`CASE → DL → RULE`). Сейчас есть один кейс (CASE-002), шаблон, скрипты создания и валидации, CI. Главное архитектурное давление — понять, достаточно ли текущей структуры для того, чтобы система росла, и что сломается первым.

## ArchGate v3: оценка по ЭМОГССБ

> **Примечание:** пользователь не указал явно критичные характеристики и альтернативы. Ниже использованы предположения на основе назначения репозитория. Для принятия решения рекомендуется явно подтвердить выбор критичных характеристик.

### Предполагаемые критичные характеристики

1. **Эволюционируемость** — система должна легко адаптироваться под новые типы кейсов, полей, категорий.
2. **Генеративность** — кейсы должны порождать решения и правила, а не копиться как архив.

### Альтернативы

Явно рассмотренные альтернативы не указаны. Это снижает уверенность оценки. Возможные альтернативы для сравнения:
- A. Файловое markdown-хранилище (текущий вариант).
- B. Реляционная БД с веб-формой ввода.
- C. Гибрид: markdown + индекс (JSON/YAML) + поиск.

| Характеристика | Вопрос | Статус | Обоснование |
|----------------|--------|--------|-------------|
| **Э**волюционируемость | Что сломается при изменении шаблона кейса? | ⚠️ | Шаблон `TEMPLATE-CASE.md` меняется вручную; старые кейсы не мигрируют автоматически. Новые поля добавляются без schema versioning. |
| **М**асштабируемость | Что будет при 10x росте числа кейсов? | ✅ | Файловая система + git справятся с сотнями кейсов. Поиск через `grep`/`rg` останется быстрым. Проблемы начнутся при тысячах кейсов (индекс, поиск). |
| **О**бучаемость | Сколько читать, чтобы начать? | ⚠️ | README и шаблон понятны, но разграничение `raw/`, `charts/`, `reports/`, `scripts/`, `Rations/` не полностью документировано. Новый пользователь может положить файлы не туда. |
| **Г**енеративность | Создаёт ли система платформу для правил? | ⚠️ | Потенциал есть (`CASE → DL → RULE`), но механизм извлечения решений и связи с `PACK-cattle-science` не автоматизирован. Кейсы пока живут изолированно. |
| **С**корость | Быстро ли создавать и находить кейс? | ✅ | `case-new.sh` создаёт скелет за секунды. Поиск по markdown быстрый. CI-валидация лёгкая. |
| **С**овременность | Что упущено из современных подходов? | ⚠️ | Хорошо: markdown + git + агенто-читаемость. Плохо: нет автоматического FPF-контекста (`fpf_context` пуст в CASE-002), нет структурированного индекса, нет связи с онтологией `PACK-cattle-science`. |
| **Б**езопасность | Какие угрозы? | ⚠️ | Нет PII / платёжных данных, но есть коммерчески чувствимые данные ферм (названия, удои, рационы). Репозиторий git не имеет встроенного контроля доступа. Нет шифрования at-rest для сырых файлов. |

## Конкретные предложения по улучшению

### 1. Версионирование схемы кейса

Добавить в `TEMPLATE-CASE.md` поле:

```yaml
schema_version: "1.0"
```

И создать скрипт `scripts/case-migrate.sh`, который при изменении `schema_version` предлагает миграцию старых кейсов.

### 2. Машиночитаемый индекс кейсов

Создать `cases/index.yaml` или генерировать `cases/README.md` автоматически. В индексе:
- case_id
- farm
- category
- status
- tags
- dl_ref
- rule_refs

Это ускорит поиск и даст обзор без `grep`.

### 3. Автоматическое заполнение `fpf_context`

Подключить `fpf_reference` при создании кейса. Если MCP недоступен — добавить fallback: по `category` и `tags` подбирать релевантные паттерны из `PACK-cattle-science`.

### 4. Связь с `PACK-cattle-science`

В шаблоне кейса добавить раздел:

```markdown
## Связь с Pack
- related_entities: []
- related_methods: []
- related_sota: []
```

И скрипт, который проверяет, что хотя бы одна связь заполнена перед переходом в статус `processed`.

### 5. Политика хранения сырых данных

Добавить в `AGENTS.md`:
- максимальный размер файла в `raw/` (например, 10 МБ);
- большие файлы — во внешнее хранилище со ссылкой в `raw/README.md`;
- запрет на хранение персональных данных в `raw/`.

### 6. Pre-commit hook

Добавить `.githooks/pre-commit`, который запускает `case-validate.sh --all` перед коммитом. Это усилит gate локально.

### 7. Периодический review-спринт

Ввести статус `reviewed` и правило: каждый кейс в статусе `processed` должен быть рассмотрен в течение 30 дней на предмет извлечения DL/RULE.

### 8. Фиксация зависимостей скриптов

Добавить `scripts/requirements.txt` с версиями `pandas`, `matplotlib` и т.д. для воспроизводимости графиков.

**Создать Architecture Decision Record (ADR) или Decision Layer (DL) на базе этой оценки.**

Конкретно:
1. Утвердить, что `DS-cattle-cases` остаётся файловым markdown-репозиторием (не переходить на БД).
2. Принять ADR о внедрении `schema_version` и индекса кейсов.
3. Реализовать первые два улучшения: `schema_version` + `cases/index.yaml`.

## Итоговый вердикт

**Решение проходит АрхГейт** с оговорками.

- **Сильные стороны:** простота, скорость, масштабируемость для текущих объёмов, хорошая база для агентной работы.
- **Слабые стороны:** нет версионирования схемы, слабая генеративность (CASE → DL → RULE не автоматизирован), неполная безопасность коммерческих данных, отсутствие срока пересмотра.
- **Главный риск:** через несколько месяцев кейсы начнут расходиться по структуре и застопорят извлечение правил.
- **Главное улучшение:** ввести `schema_version` + машиночитаемый индекс + периодический review процесс.
