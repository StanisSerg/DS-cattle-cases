# ADR-001: Архитектура хранения кейсов в DS-cattle-cases

## Контекст

Репозиторий `DS-cattle-cases` создан как Data Space для канонических сырых кейсов по крупному рогатому скоту. Конвейер: `CASE → DL → RULE`. После ArchGate + FPF оценки (`docs/architecture-evaluation.md`) выявлены риски:

- старые кейсы расходятся по структуре при изменении шаблона;
- нет централизованного индекса кейсов;
- связь с `PACK-cattle-science` не формализована;
- нет срока пересмотра кейса.

## Решение

### 1. Оставить файловую markdown-архитектуру

Не переходить на реляционную БД или веб-приложение. Текущий объём (единицы кейсов) не требует БД, а markdown + git даёт:
- агенто-читаемость;
- версионирование из коробки;
- простоту редактирования;
- отсутствие vendor lock-in.

### 2. Ввести `schema_version` в frontmatter кейса

```yaml
schema_version: "1.0"
```

- Текущая версия: `1.0`.
- При изменении шаблона версия инкрементируется.
- `case-validate.sh` проверяет поддерживаемую версию.

### 3. Создать машиночитаемый индекс `cases/index.yaml`

Индекс содержит:
- `case_id`, `slug`, `farm`, `author`, `category`, `tags`;
- `status`, `dl_ref`, `rule_refs`;
- `created`, `updated`, `review_date`.

`case-new.sh` автоматически добавляет новый кейс в индекс.

### 4. Добавить статус `reviewed` и поле `review_date`

Статусы теперь: `raw` → `processed` → `reviewed` → `archived`.

- `reviewed` означает, что кейс проанализирован на предмет извлечения DL/RULE.
- `review_date` по умолчанию = `created + 30 дней`.

### 5. Усилить локальную валидацию

`case-validate.sh` теперь проверяет:
- наличие `schema_version`;
- поддерживаемую версию схемы;
- консистентность `cases/index.yaml`.

## Альтернативы

| Вариант | Почему отклонён |
|---------|-----------------|
| Перейти на реляционную БД | Избыточно для текущего объёма. Усложняет агентную работу и требует хостинг. |
| Сделать веб-форму ввода | Дорого в разработке и поддержке. Теряется git-версионирование. |
| Оставить всё как есть | Оценка показала 5 предупреждений и 1 ❌ в DRR Adequacy. Без изменений риск застоя системы высокий. |

## Последствия

**Плюсы:**
- Кейсы версионируются по схеме.
- Появляется обзорный индекс.
- Проще находить кейсы и отслеживать их связь с правилами.
- Закладывается основа для автоматической связи с `PACK-cattle-science`.

**Минусы / accepted losses:**
- Нужно поддерживать `cases/index.yaml` в актуальном состоянии.
- При изменении схемы потребуется скрипт миграции старых кейсов.
- Добавляет 1 поле в шаблон — небольшое усложнение для автора кейса.

## Следующие шаги

1. Реализовать скрипт миграции `case-migrate.sh` на случай `schema_version > 1.0`.
2. Добавить раздел связи с `PACK-cattle-science` (`related_entities`, `related_methods`, `related_sota`).
3. Настроить pre-commit hook с `case-validate.sh --all`.
4. Ввести периодический review-спринт для перехода кейсов в статус `reviewed`.
